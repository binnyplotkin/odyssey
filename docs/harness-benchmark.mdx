# Odyssey World Simulation Harness Benchmark

This benchmark compares the full Odyssey world simulation harness, not just
the underlying language model. A column is a runnable harness configuration:
model, prompt snapshot, retrieval behavior, world/scene orchestration,
character and identity management, voice stack, endpointing, runtime code, and
suite versions. A row is a capability or operating constraint we care about.

## Axes

Use these as the canonical row families for v1:

| Family | Source | Primary signal |
|---|---|---|
| Character quality | `eval_runs.avg_overall` | Judge composite, normalized to 0-100 |
| Identity fidelity | `eval_probe_results` identity + trait probes | Difficulty-weighted probe overall |
| Voice style | Judge `voice` dimension | Difficulty-weighted dimension average |
| Grounded factuality | Judge `factual` dimension | Difficulty-weighted dimension average |
| Scope + safety | scope, deflect, jailbreak probes | Difficulty-weighted pass/overall blend |
| Continuity | edge + frame probes | Difficulty-weighted probe overall proxy |
| Agency | Sonar `agency-baseline` + judged rows | Live conversation control and initiative |
| Context activation | Sonar traced run + context score rows | Retrieval, curation, cache, and context injection |
| Regression pass | `passed / total` | Pass rate |
| Text latency | `eval_runs.avg_latency_ms` | Lower is better |
| Voice latency | Sonar `voice-to-voice` p50 | Lower is better |
| Endpointing | Sonar cutoff rate | `1 - cutoff_rate` |
| Reliability | eval and Sonar errors | Error-free rate |
| Cost efficiency | eval + Sonar estimated cost | Lower is better |

## Composite

The first report command uses this weighted score:

| Dimension | Weight |
|---|---:|
| Character quality | 20% |
| Grounded factuality | 15% |
| Scope + safety | 10% |
| Continuity | 10% |
| Agency | 15% |
| Context activation | 10% |
| Voice latency | 10% |
| Endpointing | 5% |
| Reliability | 3% |
| Cost efficiency | 2% |

When a dimension is missing, the report reweights across available dimensions
and prints coverage. A score with low coverage is useful for triage, but should
not be treated as a decision-grade benchmark.

## Metric Semantics

Cells are **measured scores on a slice**, not declarations of mastery. A
`100%` means "the harness cleared the current probes in this family." It does
not mean the row is done.

To keep room for growth, every quality cell prints its basis:

```txt
100.0%
7p · 5 hard / 2 adversarial · low confidence · saturated
```

The basis line means:

- `7p` — number of probes behind the score.
- `5 hard / 2 adversarial` — current difficulty mix.
- `low confidence` — sample-size confidence, not model confidence.
- `saturated` — the score is at or above 95%, so the row needs harder or
  hidden probes before it should be treated as decision-grade.

Difficulty is currently inferred from probe category until probe definitions
carry explicit labels:

| Difficulty | Weight | Current categories |
|---|---:|---|
| basic | 1x | identity |
| medium | 2x | trait, frame |
| hard | 3x | scope, deflect, edge |
| adversarial | 4x | jailbreak |

Confidence is based on sample size:

| Probe / turn count | Confidence |
|---:|---|
| `< 12` | low |
| `12-29` | medium |
| `>= 30` | high |

## Agency

**Agency** is the harness's ability to actively participate in a live world
conversation: manage turns, accept interruption/correction, engage a low-signal
user, repair misreads, choose a useful next step, and keep the simulated world
moving.

Agency is intentionally not a synonym for autonomy or tool use. In this
benchmark it means **live conversation control and initiative**.

The first Sonar suite is:

```sh
npm run sonar -- run --suite agency-baseline --label "agency baseline"
npm run sonar -- judge-agency --latest
```

The suite exercises:

| Dimension | What good looks like |
|---|---|
| Turn-taking | Stops, resumes, and hands the floor back at natural moments |
| Interruptability | Accepts a user correction or stop request without continuing the old track |
| Engagement | Responds with specificity and energy instead of generic acknowledgement |
| Initiative | Offers the next useful question, choice, or action when the user is unsure |
| Repair | Recovers after confusion, partial information, or a changed intent |
| Goal persistence | Maintains the larger session purpose across turns |
| World responsiveness | Advances scene state, character presence, and narrator mediation appropriately |

The current runner can test **turn-level** interruption/correction, but not true
mid-agent barge-in. True barge-in needs a future overlapping-audio Sonar runner
that starts user speech while agent audio is still streaming and scores stop
latency plus recovery quality.

Agency is scored only when a judged row exists in
`evals/sonar/agency-scores.jsonl`. The judge command reads a local
`.sonar/runs/...agency-baseline...json` record, evaluates the full transcript,
and upserts the score row:

```sh
npm run sonar -- judge-agency --latest
npm run sonar -- judge-agency --run-id <run-prefix>
npm run sonar -- judge-agency --file .sonar/runs/<file>.json --dry-run
```

The benchmark does not use latency as a proxy for Agency because a fast answer
can still fail to engage, repair, or drive the conversation.

Each JSONL row is keyed by Sonar `runId`:

```json
{"runId":"...","at":"2026-06-16T00:00:00.000Z","sonarVersion":"0.1.0","suite":"agency-baseline","suiteVersion":"0.1.0","model":"gpt-oss-120b","turns":10,"judge":"manual-v0","dimensions":{"turnTaking":80,"interruptability":70,"engagement":85,"initiative":75,"repair":70,"goalPersistence":80,"worldResponsiveness":78},"notes":"Short rationale for failures and regressions."}
```

The CLI judge emits dimension scores on a 0-100 scale, applies severe-failure
penalties capped at 30 points, and writes a final `score`. Scores should be
calibrated like model evals: leave room above the current best run by adding
harder hidden scenarios when a dimension saturates.

## Context Activation

**Context Activation** is the benchmark family for content management,
knowledge graph use, and per-term context injection. It asks whether the
harness can activate the right world context for the current turn quickly,
selectively, and within budget.

The first scorer is deterministic and runs on Sonar trace evidence:

```sh
npm run sonar -- run --suite context-activation-baseline --label "context baseline"
npm run sonar -- score-context --latest
npm run sonar -- score-context --run-id <run-prefix>
npm run sonar -- score-context --file .sonar/runs/<file>.json --dry-run
```

It writes `evals/sonar/context-activation-scores.jsonl`. The benchmark joins a
row only when that file contains a score for the matching Sonar run.

| Dimension | What it measures |
|---|---|
| Context availability | How often traced turns receive attached context |
| Retrieval recall | Expected page slugs selected for the turn |
| Retrieval precision | Selected pages that match expected slugs, minus forbidden-page drift |
| Curation selectivity | Whether the curator selects a focused page set |
| Token efficiency | Whether context stays inside the token budget |
| Cache effectiveness | Cache hit rate after the first turn in a session |
| Retrieval latency | Retrieval p50 |
| Curator latency | Curator p50 |
| Context attach latency | Request received to context attached p50 |

`context-activation-baseline` carries gold expected wiki pages and must-not
inject pages per turn, so new runs score recall and precision directly. Older
trace-only runs remain scoreable by proxy for historical continuity.

## Provenance

Every comparable column should carry:

- `character_id`
- `character_snapshot`
- `config_hash`
- `eval_suite` and `suite_version`
- `judge_model`
- `model`, `temperature`, `max_tokens`
- retrieval/context mode when applicable
- `sonar_version`
- Sonar suite/version
- Agency score row and judge id when present
- Context Activation score row when present
- observed voice/STT/TTS providers
- `git_sha` plus dirty flag
- human-readable experiment `label`

The existing eval store captures the character snapshot and config hash. Sonar
captures methodology version, suite version, git state, observed model/provider,
latency spans, endpointing, errors, and cost in `evals/sonar/ledger.jsonl`.

## Commands

Run the underlying suites first:

```sh
npm run eval -- abraham
npm run sonar -- run --suite voice-baseline --label "baseline"
npm run sonar -- run --suite agency-baseline --label "agency baseline"
npm run sonar -- run --suite context-activation-baseline --label "context baseline"
npm run sonar -- score-context --latest
npm run sonar -- run --suite real-endpointing --audio-rt-ws ws://127.0.0.1:8089/api/asr-streaming
```

Then render the joined benchmark:

```sh
npm run benchmark -- report --character abraham
npm run benchmark -- report --character abraham --limit 8
npm run benchmark -- weights
```

## Run groups

Pass `--run-group <id>` (or set `ODYSSEY_RUN_GROUP`) on every Sonar launch in an
experiment and the id is stamped on the run record and ledger row. Then
`npm run benchmark -- report --run-group <id>` scopes **all** Sonar evidence —
voice, agency, context activation, endpointing — to that one experiment, so a
column's runtime rows come from the same launch instead of the nearest run that
merely shares an observed model.

```sh
export ODYSSEY_RUN_GROUP=abraham-2026-06-17-flux
npm run sonar -- run --suite voice-baseline --label "flux endpointer"
npm run sonar -- run --suite agency-baseline --label "flux endpointer"
npm run sonar -- run --suite context-activation-baseline --label "flux endpointer"
npm run benchmark -- report --character abraham --run-group abraham-2026-06-17-flux
```

## SLO attainment

Latency rows headline `p50`/`p95`/`p99`, but percentiles only describe the
distribution's shape — they don't say how many turns actually cleared the bar.
Each run also reports **voice-to-voice SLO attainment**: the share of turns at
or under a target (default 1500ms; override per run with `--slo-ms` or per suite
with `SonarSuite.v2vSloMs`). The ledger carries `v2vSloMs`/`v2vSloPct`, the
progression table shows it as `<pct>%@<target>`, and the benchmark's Voice
latency basis prints `p50 · p99 · SLO attainment`.

The 1500ms default is a near-term engineering target (roughly the production
commit hold we plan to cut), not the human-conversation ideal. As the pipeline
approaches the sub-second goal, tighten the target so attainment keeps pulling
rather than saturating.

## TTS A/B (worked example)

Compare the current Pocket binding against hosted ElevenLabs streaming on the
same run group, at the new 40-sample `voice-baseline`. Both arms route through
the `--tts-voice` override path so the only variable is the voice itself
(control passes the character's own slug rather than omitting the flag):

```sh
# dev server up + ODYSSEY_ADMIN_COOKIE set; both arms share one run group
export RG=tts-ab-$(date +%Y%m%d)

# arm A — control: current Pocket binding (pocket_tts:abraham)
npm run sonar -- run --suite voice-baseline --tts-voice abraham \
  --run-group "$RG" --label "tts A · pocket (control)"

# arm B — hosted ElevenLabs streaming
npm run sonar -- run --suite voice-baseline --tts-voice liam \
  --run-group "$RG" --label "tts B · elevenlabs streaming"

# read both arms side by side
npm run sonar -- report --suite voice-baseline --last 2
npm run benchmark -- report --character abraham --run-group "$RG"
```

Watch `server.tts.ttfa` and `vs.ttfa` for the first-audio win, and confirm
`v2vSloPct` (share of turns under the 1500ms target) moves with it. Each arm is
40 turns of real STT+LLM+TTS (~7 min, a few cents); run them back to back so
network and provider conditions match.

## Current Limitation

The eval side does not yet carry the run-group id, so eval quality still joins
to Sonar runtime by observed model even under `--run-group`. The run group makes
the *Sonar* cross-suite join exact today; making the *eval↔Sonar* join exact is
the remaining step (stamp the same id on eval launches and filter eval runs by
it).
